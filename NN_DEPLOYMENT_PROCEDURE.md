# Neural Network Deployment Procedure (VOXL2 + ROS2)

This document describes a minimal workflow for training and deploying ROS2 Python inference nodes in this repository using:

1. Native PyTorch runtime on VOXL2
2. TFLite runtime on VOXL2 (converted from the same model via ONNX -> onnx2tf)
3. A single ROS2 Python node (inference script wrapped as one ROS2 package)

---

## Scope and Assumptions

- Workstation: x86_64 Linux with Docker + buildx
- Drone: VOXL2 (aarch64) with Docker
- Repo path on workstation: this repository root
- ROS2 distro: Humble
- Inference node: one Python ROS2 node in `ros2_ws/src`

Model conversion is done on the workstation. VOXL2 only runs inference.

---

## 1) Minimal Dependency Strategy

To reduce complexity, keep only dependencies required for:
- ROS2 Python package build/run
- PyTorch inference
- TFLite inference

Avoid adding unrelated CV/SLAM/GUI dependencies unless needed by your actual model pipeline.

---

## 2) Create a Single ROS2 Python Package

Inside `ros2_ws/src` create one package (example: `nn_inference_node`) with:
- `package.xml`
- `CMakeLists.txt`
- `nn_inference_node/__init__.py`
- `nn_inference_node/node.py`
- `nn_inference_node/runtime_models/`

Recommended runtime parameter in node:
- `backend`: `pytorch` or `tflite`
- `model_path`: absolute path in container (for example `/ros2_ws/src/nn_inference_node/nn_inference_node/models/model.pt` or `.tflite`)

Node behavior (minimal):
- subscribe to one input topic
- preprocess tensor
- run inference (`torch` or `tflite_runtime.interpreter`)
- publish output topic

---

### EXTENSION

Create ros2 package running two nodes. One that runs a tflite yolo model for inference on GPU, and one node that runs parallel pytorch training on CPU (The Voxl NPU is not cuda compatible). 
It would have to have on-board datasets for training since it is suporvised. This is just to test the feasability of running on board training. Will be usefull knowledge for later unsuporvised training of models.


## 3) Train on Workstation (PyTorch)

Train outside VOXL deploy path (workstation only):

1. Train model using your normal training loop.
2. Save native checkpoint (`model.pt` / `model.pth`) — this is the primary artifact.

Keep training artifacts versioned, for example:
- `models/model_v1.pt`
- `models/model_v1.onnx`   ← generated by conversion script
- `models/model_v1.tflite` ← generated by conversion script

---

## 4) Convert PyTorch -> ONNX -> TFLite (onnx2tf pipeline)

This step is **only required for the `tflite` backend**. For the `pytorch` backend, the
`.pt` file from section 3 is used directly — skip this section.

Run on workstation (not on VOXL2 runtime):

### Step 1 — Prepare source artifacts

- Place PyTorch checkpoint at `models/yolov8n.pt` (or adjust `MODEL_NAME` and `DUMMY_INPUT_SHAPE` in `scripts/pt_onnx_tflite_pipeline.py` depending on your model).
- Place calibration images under `models/quantization_data/Images/`.

### Step 2 — Run conversion pipeline

The repository script handles:
- PyTorch -> ONNX export
- ONNX -> TFLite conversion via `onnx2tf`
- INT8 conversion retries (`*_auto.json` usage)
- optional validation phase

```bash
# from repo root
python3 scripts/pt_onnx_tflite_pipeline.py
```

Optional validation pass:

```bash
python3 scripts/pt_onnx_tflite_pipeline.py --validate
```

### Step 3 — Check outputs

Generated artifacts are written under:
- `models/Model.onnx`
- `models/saved_model/Model_XX.tflite`

Typical TFLite outputs include float and quantized variants (names depend on onnx2tf version).

### Step 4 — Place model in the ROS2 package

Copy the final artifacts into the node's models directory depending on target backend.
EXAMPLE:

| Backend    | File required                                              |
|------------|------------------------------------------------------------|
| `pytorch`  | `ros2_ws/src/nn_inference_node/nn_inference_node/models/model.pt`      |
| `tflite`   | `ros2_ws/src/nn_inference_node/nn_inference_node/models/model.tflite`  |

Both can coexist in the same folder; the node selects via the `backend` parameter.

---

## 5) Build and Deploy (Repository Workflow)

From repository root:

1. Build ARM64 dev image:
   - `make build-cross`
2. Build ROS2 workspace in ARM64 container:
   - `make build-ws-cross`
3. Build ARM64 runtime image:
   - `make build-runtime`
4. Deploy workspace + runtime compose/config to VOXL:
   - `make deploy`
5. Export + transfer + load runtime image on VOXL:
   - `make deploy-image`
6. Start runtime container:
   - `make voxl-start`
7. Inspect logs:
   - `make voxl-logs`

---

## 6) Run Native PyTorch on VOXL2

Prerequisites: `model.pt` present in the deployed package models directory (section 4, no
conversion needed).

1. Launch node with `backend:=pytorch` and `model_path` pointing to the `.pt` file.
2. Confirm in logs that `import torch` succeeds and the model loads once at startup.
3. Measure FPS/latency; reduce input resolution or batch size if needed.

Notes:
- CPU-only inference on VOXL is expected unless explicit acceleration support is present.
- If ARM64 PyTorch wheel compatibility becomes an issue, prefer the TFLite path.

---

## 7) Run TFLite on VOXL2

Prerequisites: `model.tflite` generated via the `PyTorch → ONNX → onnx2tf` pipeline in
section 4 and present in the deployed package models directory.

1. Launch node with `backend:=tflite` and `model_path` pointing to the `.tflite` file.
2. Validate interpreter allocation and first inference in startup logs.
3. Compare accuracy/speed against PyTorch on same test samples if both are available.

TFLite is usually the safer lightweight runtime path on constrained ARM platforms.

---

## 8) Suggested Validation Checklist

Before flight/field usage:

- Node starts in container with no import errors
- Model file found at configured path
- Input topic and output topic active
- Inference rate stable for at least 5 minutes
- CPU and memory usage acceptable
- Recovery behavior confirmed (node restart, container restart)

---

## 9) Quick Command Set

- Build cross image: `make build-cross`
- Cross build workspace: `make build-ws-cross`
- Build runtime image: `make build-runtime`
- Deploy files: `make deploy`
- Deploy runtime image: `make deploy-image`
- Start drone container: `make voxl-start`
- Logs: `make voxl-logs`
- Stop container: `make voxl-stop`

---

## 10) Practical Recommendation

Start with TFLite first for deployment confidence and lower runtime overhead, then enable native PyTorch as a second backend in the same node. This keeps one ROS interface while allowing backend switching by parameter.
